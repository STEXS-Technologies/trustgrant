# TrustGrant Ownership Authority Transitions

## 1. Purpose

This document defines the concrete ownership-authority transition profile for
TrustGrant-aware systems.

The goal is to support cases where:
- a resource or canonical resource lineage is transferred from one authority to another
- the successor authority must receive future delegation and monetization rights
- the canonical identity of already-existing resources must remain stable

This profile exists because a normal operational TrustGrant is not enough to model full
ownership transfer safely.

## 2. Why This Is Not Just a Broad Mint Grant

An ordinary TrustGrant delegates operational authority.

It can authorize:
- `recognize`
- `mint`
- service-defined operations

But an ordinary TrustGrant should not silently rewrite:
- who is the active owning authority of a canonical resource lineage
- who may issue future TrustGrants on that lineage as owner
- who receives redirected royalty or continuity rights

Ownership transfer is stronger than operational delegation.
It changes future authority over a lineage.
That requires a dedicated proof/profile.

## 3. Core Model

Ownership transition uses three distinct concepts:

- **Origin authority**
  - the authority under which the canonical resource lineage was originally created
  - remains part of historical identity and provenance
- **Active owning authority**
  - the authority currently recognized as owning the lineage for future delegation and
    settlement
- **Successor authority**
  - the authority that becomes the active owning authority after a valid transition

The key rule is:
- canonical identity does not change when ownership changes

That means a transfer does not clone or rename the resource lineage.
It only changes who may act as the current owning authority for future rights
management.

`transition_id` is the exact protocol identity of the transition proof itself.
It must be generated by transition-preparation tooling before the signed transition
payload is finalized, because the signed proof needs a stable document identity.
Verifiers may create local storage keys, but they must not replace or rewrite
`transition_id`.

Canonical v0 format:

```text
transition_id        = tgt_<uuid>
transition_series_id = tgts_<uuid>
```

## 4. Invariants

The transition profile must preserve these invariants:

- a canonical resource lineage has at most one active owning authority at a time
- a valid transition must start from the currently active owning authority
- the successor authority must explicitly accept the transfer
- accepted ownership transfer must not rewrite historical provenance
- future TrustGrants on the transferred lineage may be issued by the successor authority
  only after the transition becomes active
- the predecessor authority must not continue issuing owner-level grants on that lineage
  after the transition becomes active, unless separately delegated by the successor
  authority
- if two competing transitions claim active ownership for the same lineage, verification
  fails closed until the conflict is resolved

## 5. Transition Record

The concrete transition proof is a dedicated transfer record, not a normal resource-use
grant.

The v0 implementation should treat the following field names as the machine-contract
baseline. The JSON schema artifact is
[TRUSTGRANT_OWNERSHIP_AUTHORITY_TRANSITION_V0_SCHEMA.json](TRUSTGRANT_OWNERSHIP_AUTHORITY_TRANSITION_V0_SCHEMA.json).

Canonical shape:

```json
{
  "transition_id": "tgt_550e8400-e29b-41d4-a716-446655440200",
  "version": 0,
  "transition_series_id": "tgts_550e8400-e29b-41d4-a716-446655440300",
  "revision": 1,
  "supersedes_transition_id": null,
  "origin_authority": "authority_id",
  "from_authority": "authority_id",
  "to_authority": "authority_id",
  "canonical_resource_scope": {
    "types": {
      "item": {
        "all": false,
        "allow": [
          {
            "kind": "id",
            "all": false,
            "values": ["canonical_item_1", "canonical_item_2"],
            "expressions": null
          }
        ],
        "deny": null
      }
    }
  },
  "global_constraints": {
    "time": {
      "not_before": "2026-01-01T00:00:00Z",
      "not_after": "2026-02-01T00:00:00Z"
    }
  },
  "effective_at": "2026-01-10T00:00:00Z",
  "predecessor_signature": {
    "key_id": "owner-2026-01",
    "signature": "base64(signature)"
  },
  "successor_acceptance": {
    "accepted_at": "2026-01-05T10:00:00Z",
    "key_id": "successor-2026-01",
    "signature": "base64(signature)"
  }
}
```

This shape is now the v0 machine-contract baseline for implementation.
Later protocol revisions may extend it, but v0 consumers should not rename these fields
ad hoc.

The transition proof itself is the machine-contract payload.
Transport APIs may wrap it in request envelopes, but they must not mutate or rename the
signed transition fields.

## 6. Validation Rules

Structural and semantic validation must enforce:

- `from_authority != to_authority`
- `origin_authority` is explicit
- `canonical_resource_scope` is finite and explicit
- `canonical_resource_scope.all = true` is forbidden for this profile
- expressions are forbidden in transition scope for v0
- `effective_at` must fall within the allowed time window if one is declared
- `supersedes_transition_id`, when present, must refer to an older transition in the
  same transition series
- predecessor signature must cover the exact transition payload
- successor acceptance must cover the exact accepted transition payload or its canonical
  digest
- `from_authority` must be the currently active owning authority at verification time
- no overlapping accepted transitions may leave the same canonical lineage with multiple
  active owners

## 7. Acceptance Rules

Evidence is sufficient only when all of these hold:

1. predecessor signature verifies
2. predecessor is the current active owning authority for the affected canonical
   resource scope
3. successor acceptance signature verifies
4. transition scope resolves to explicit canonical resources or namespaces allowed by
   local policy
5. no conflicting accepted transition exists for the same scope and activation window
6. the verifier's local trust policy accepts both authorities and the transfer profile

For deployments that maintain an ownership registry, the transition is only considered
active after it has been registered and accepted by that registry.

## 8. Runtime Consequences

Once active:

- the successor authority becomes the active owning authority for the transferred
  canonical lineage
- royalty and continuity rights redirect to the successor authority
- the predecessor authority stops being the default owner-level issuer for that lineage
- future TrustGrants for the transferred lineage may be issued by the successor
  authority

Important nuance:
- successor-issued grants must not rewrite canonical resource identity
- the original origin authority remains part of historical provenance

That means verifier inputs must distinguish:
- historical origin identity
- current owning authority

## 9. Future Grant Issuance After Transfer

After transfer, the successor authority may issue new TrustGrants for the transferred
lineage, but only with an ownership-transition proof chain.

The required model is:

- canonical resource identity remains tied to the original lineage
- successor authority signs the new TrustGrant as the current owning authority
- verifier receives or resolves the active ownership-transition proof chain
- verifier confirms the signer is the active owning authority for that lineage

This avoids the broken model where ownership transfer would require either:
- mutating historical resource identity, or
- pretending the predecessor still owns the lineage forever

## 10. Conflict Resolution

If multiple transition claims exist:

- fail closed by default
- require explicit supersession metadata or manual resolution
- never pick a winner implicitly based only on timestamp

A safe default is:
- accepted active transition wins only when it cleanly supersedes the previously
  accepted transition or extends the chain from the currently active owner

## 11. Closure and No-Successor Cases

If no valid successor authority exists:

- ownership transfer does not occur
- no new authority gains owner-level grant issuance power
- continuity-mode platform policy may still apply under local deployment rules

This is the basis for solidarity-fee or similar local fallback policies when the
original authority no longer exists and no valid transfer was recorded.

## 12. Implementation Guidance

The `trustgrant` crate should treat ownership transitions as verifier inputs adjacent to
TrustGrant verification, not as ad hoc platform-side exceptions.

That means implementation should expose:

- explicit transition record types
- explicit predecessor and successor proof verification
- normalized active-owning-authority state
- trait boundaries for loading transition proofs
- fail-closed conflict handling

Hot-path evaluation should consume normalized post-transition ownership state rather
than recomputing transition chains on every request.

## 12.1 Transition Ledger and Proof-Chain Storage

Systems adopting this profile should persist more than the currently active owner
pointer.

At minimum, an append-only transition ledger or immutable-enough proof-chain store
should retain:
- submitted transition proofs
- acceptance or rejection outcomes
- conflict markers
- supersession relationships
- activation timestamps
- linkage from affected canonical lineage to the accepted transition chain

This storage must preserve both:
- canonical protocol identity (`transition_id`, `transition_series_id`)
- any local storage record key used by the adopting system

The current active-owning-authority state may be materialized separately for fast reads,
but the accepted/rejected transition history must remain reconstructible without
guessing from only the latest snapshot.

## 12.2 Logical Persistence Model

Deployments implementing this profile should keep four logical persistence layers
distinct even if they share physical infrastructure:

1. **Transition Proof Store**
   - immutable retained transition payload or canonical digest plus blob reference
   - keyed by canonical `transition_id`

2. **Transition Status Projection**
   - mutable current-state record keyed by canonical `transition_id`
   - stores:
     - `transition_series_id`
     - current transition status
     - effective-at timestamp
     - predecessor and successor authority summary

3. **Active Owner Projection**
   - current active owning authority snapshot per canonical lineage
   - keyed by canonical lineage identity, not by local row key
   - references the currently accepted `transition_id` when ownership changed

4. **Transition Audit Ledger**
   - append-only acceptance, rejection, supersession, and conflict history
   - keyed by canonical `transition_id` plus canonical lineage reference

This split exists so implementations do not overload one mutable row with:
- raw proof storage
- current acceptance status
- current active owner snapshot
- historical audit history

## 12.3 Protocol IDs vs Local Record IDs

`transition_id` and `transition_series_id` are canonical protocol identifiers generated
before signing.

An adopting system may create local row keys, blob handles, or registry record IDs, but:
- those local identifiers must remain implementation-local
- they must not replace protocol identity in verifier logic
- they must not replace protocol identity in cross-system export or public APIs unless
  explicitly introduced as separate local handles
